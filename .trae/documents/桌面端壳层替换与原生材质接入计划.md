# 桌面端壳层替换与原生材质接入计划（desktop/common + macOS Liquid Glass + Windows Fluent）

## Summary（摘要）
在当前分支既有 `UiRouteHost` 分流基础上，完成两件事：
1) 将 `desktop/common` 从“复用 `AppShellPage`”逐步替换为真正的**桌面壳层**（侧栏 Sidebar + 顶部工具区 Top Tool Area + 内容区 Content），并沉淀键盘导航/焦点态/hover/减少动效等桌面基础交互；
2) 开始接入**macOS 原生 Liquid Glass 壳层**（`NSVisualEffectView` + 原生 Sidebar/Toolbar + MethodChannel 驱动 Flutter 内容区）与 **Windows 11 Fluent 窗口材质**（Mica 常驻 + Acrylic 临时，基于 `DwmSetWindowAttribute` 设置 `DWMWA_SYSTEMBACKDROP_TYPE`），为后续“原生壳层驱动导航与命令”留好通道协议与扩展点。

> 设计/交互约束：遵循 ui-ux-pro-max 的高优先级原则（a11y、键盘导航、focus ring、对比度、hover、减少动效），且所有新增用户可见文案必须走 `context.l10n.xxx`（不得硬编码）。

---

## Current State Analysis（现状分析，基于代码库）
### 1) desktop/common 仍在复用 AppShellPage
- `lib/ui/desktop/common/app/desktop_shell_page.dart` 目前直接返回 `AppShellPage`，桌面壳层结构并未独立。
- `lib/ui/desktop/macos/app/macos_shell_content_page.dart` 与 `lib/ui/desktop/windows/app/windows_shell_content_page.dart` 目前均回退到 `DesktopShellPage`，尚未与原生壳层形成“内容区 + channel 驱动”闭环。

### 2) 路由分流已有入口但覆盖面有限
- `lib/ui/routing/ui_route_host.dart` 会在 `build()` 内解析 `UiTarget` 并对 `/home` 做平台分流。
- `lib/config/app_router.dart` 当前只在 `AppRoutes.home` 使用 `UiRouteHost`；例如 `AppRoutes.files` 仍可直接构建 `AppShellPage`，这会绕开桌面/原生壳层分流。

### 3) macOS 原生与外观桥接已有基础，但未实现“原生侧栏+toolbar”
- `macos/Runner/MainFlutterWindow.swift` 已存在窗口外观调整与 `onepanel/macos_appearance` channel 的上下文读取能力。
- Flutter 侧已有 `MacosAppearanceChannel` / `MacosAppearanceContextModel`。

### 4) Windows runner 目前仅处理暗色标题栏属性
- `windows/runner/win32_window.cpp` 当前主要设置 `DWMWA_USE_IMMERSIVE_DARK_MODE`，尚未设置 `DWMWA_SYSTEMBACKDROP_TYPE`（Mica/Acrylic）。

---

## Assumptions & Decisions（假设与决策）
1) **信息架构（桌面壳层）**：先采用“多模块骨架”落地（Sidebar 以模块切换为核心，结构按桌面端设计要求实现；可先覆盖 Server/Files/Settings + 预留槽位）。
2) **macOS 第一阶段目标**：实现“原生 Sidebar + 原生 Toolbar + Liquid Glass（NSVisualEffectView）”，Flutter 仅作为内容区并通过 channel 驱动导航/模块切换。
3) **Windows 材质策略**：Mica 用于常驻表面（主窗口背景/标题栏）；Acrylic 用于临时表面（右键菜单/弹窗/侧边栏等）。第一阶段优先在 runner 落地 Mica（常驻）与提供切换 Acrylic（临时）的能力（可先内部 API，后续再暴露 channel）。
4) **不改变数据层**：复用现有 `lib/api/`、`lib/data/`、`lib/core/*` 与既有 controllers/providers；桌面端仅做 UI/交互分离，不复制业务逻辑到 Widget `build()`。
5) **平台覆盖范围**：本阶段聚焦 desktop/common + macOS + Windows；Linux/Web 继续使用 desktop/common（后续再扩展）。

## Proposed Changes（改造方案：具体文件与实现步骤）

### A. desktop/common：实现真实桌面壳层（Sidebar + Top Tool Area + Content）
**目标**：替换 `DesktopShellPage` 中对 `AppShellPage` 的直接复用，形成桌面独立壳层；同时保证可复用现有模块控制器（避免重复造轮子）。

#### A1. 新建/完善桌面壳层组件（Flutter）
新增/修改文件（以当前结构为准，可按需要拆分子组件，遵守 1000 LOC 上限）：
- `lib/ui/desktop/common/app/desktop_shell_page.dart`
  - 改为桌面三段布局：左侧 `DesktopSidebar` + 顶部 `DesktopTopToolArea` + 右侧 `DesktopContentHost`。
  - 内容区建议使用 `IndexedStack`，减少切换动画并保持状态（同时尊重 `MediaQuery.disableAnimations`）。
- `lib/ui/desktop/common/widgets/desktop_sidebar.dart`
  - Sidebar 的模块列表与分组（桌面端信息架构）。
  - 键盘导航：`FocusTraversalGroup` + `Shortcuts/Actions`（上下/Enter/快捷键切换模块）。
  - hover/focus 样式：使用 `InkResponse/InkWell` 与主题色（`Theme.of(context).colorScheme`），确保 focus ring 可见。
- `lib/ui/desktop/common/widgets/desktop_top_tool_area.dart`
  - 顶部工具区：模块标题、ServerSwitcher、全局搜索入口（占位）、刷新（占位）、设置/帮助入口（占位）。
  - 所有 tooltip/label 使用 `context.l10n.xxx`。
- `lib/ui/desktop/common/widgets/desktop_content_host.dart`
  - 负责根据当前选中模块渲染内容页（见 A2 的“模块->页面工厂”复用策略）。

#### A2. 抽离“模块 -> 页面”构建为可复用工厂（避免复制 switch）
**目标**：桌面壳层、移动壳层、macOS/Windows 内容区共享同一份模块映射（只共享“页面组合”，不引入业务逻辑）。

优先复用/下沉位置（避免 UI 层循环依赖）：
- 将 `AppShellPage._buildCurrentPage(...)` 的模块映射抽到：
  - `lib/features/shell/module_page_factory.dart`（建议）或 `lib/features/shell/widgets/...` 中的独立文件
  - 工厂函数签名建议：`Widget buildModulePage(BuildContext context, ClientModule module, String? serverId)`，并保留现有 server gating 行为（NoServerSelectedState 等）。

桌面壳层通过工厂构建内容区，从而：
- 移动端 `AppShellPage` 可继续使用（保持现状）
- `desktop/common` 可替换壳层但不复制业务页面选择逻辑
- macOS/Windows 的“原生壳层驱动 Flutter 内容区”可直接驱动 moduleId/routeName，而无需维护两份映射表

#### A3. 解决“模块页自带 AppBar 与桌面顶部工具区冲突”
短期策略（可渐进）：
- 桌面壳层先渲染 Top Tool Area，模块页继续保留自身 AppBar（可接受但体验一般）。
中期策略（推荐）：
- 在 `lib/features/shell/widgets/server_aware_page_scaffold.dart` 增加“嵌入模式”参数（例如 `showAppBar`/`embedMode`），当处于桌面壳层时模块页不再渲染 AppBar，把标题与全局动作统一收敛到 Top Tool Area。

> 注意：业务逻辑仍下沉到 Provider/Controller/Service，UI 仅做展示与触发。

---

### B. 路由分流完善：避免绕开 UiRouteHost
**目标**：确保桌面端（尤其 macOS 原生壳层）不会被 `/files` 等入口路由绕开。

两种实现路线（二选一，推荐路线 1）：

#### 路线 1（推荐）：模块入口路由统一 redirect 到 `AppRoutes.home`
- 在 `lib/config/app_router.dart` 中对 `AppRoutes.files`/`containers`/`apps` 等入口路由：
  - 不再直接构建 `AppShellPage`
  - 改为构建 `UiRouteHost(settings: RouteSettings(name: AppRoutes.home, arguments: {...}))`
  - arguments 携带 `{ 'module': 'files', 'tab': 1 }` 等
- `UiRouteHost` 统一处理平台分流与模块初始化。

#### 路线 2：在 `RouteRegistry` 注册这些语义路由
- 在 app 启动时（例如 `main.dart` 或某个 routing init 位置）调用 `RouteRegistry.registerAll(...)` 注册路由表。
- 每个语义路由都提供按平台的 builder（mobile/desktop/mac/windows），避免散落 switch。

### C. macOS：原生 Liquid Glass 壳层（Sidebar + Toolbar + NSVisualEffectView）+ channel 驱动 Flutter 内容
**目标**：第一阶段即可让 macOS 体验“像原生桌面应用”，原生壳层负责导航/命令，Flutter 专注内容区。

#### C1. 原生结构改造（Swift/AppKit）
修改文件：
- `macos/Runner/MainFlutterWindow.swift`

实现要点：
- 引入一个原生容器 `NSViewController`（例如 `MainShellViewController`）作为 `contentViewController`。
- 使用 `NSSplitViewController`：
  - Sidebar pane：`NSVisualEffectView`（选择适合 sidebar 的 material；支持深色模式/减少透明/高对比度）
  - Content pane：嵌入 `FlutterViewController.view`（Flutter 内容区）
- Toolbar：使用 `NSToolbar` 或 `NSTitlebarAccessoryViewController` 在标题栏放置搜索/segmented/命令按钮等。

参考（官方文档）：
- Apple `NSVisualEffectView`：https://developer.apple.com/documentation/appkit/nsvisualeffectview
- Apple `NSTitlebarAccessoryViewController`：https://developer.apple.com/documentation/appkit/nstitlebaraccessoryviewcontroller

#### C2. Channel 协议（最小闭环 + 可扩展）
保留现有：
- `onepanel/macos_appearance`（读取 reduceTransparency/highContrast 等上下文）

新增（建议命名）：
- `onepanel/macos_shell`

Native -> Flutter（驱动内容）：
- `navigate`：`{ route: String, arguments: Map? }`（pushNamed 语义路由）
- `selectModule`：`{ moduleId: String }`（若桌面内容区采用模块切换而非路由）
- `performToolbarAction`：`{ action: String, payload?: Map }`

Flutter -> Native（同步壳层状态）：
- `setTitle`：`{ title: String }`
- `setEnabledModules`/`setHasServer`：用于 sidebar 禁用需要 server 的模块（可后置）
- `setBadges`：模块 badge（可后置）

Flutter 侧新增文件（建议）：
- `lib/ui/desktop/macos/bridge/macos_shell_channel.dart`
  - 统一 try/catch + `appLogger.*WithPackage` 日志、兜底策略（参考现有 `MacosAppearanceChannel` 的风格）。
- `lib/ui/desktop/macos/app/macos_shell_content_page.dart`
  - 变为“纯内容区容器”，监听 channel 的 module/route 指令并渲染对应内容（通过 A2 的模块页面工厂复用）。

#### C3. ui-ux-pro-max（macOS）落地点
- 键盘导航：原生 sidebar 列表必须支持上下键/Enter；toolbar 控件可 Tab 聚焦。
- 焦点态：保持系统 focus ring；Flutter 内容区在键盘输入下 focus traversal 正常。
- reduce transparency / high contrast：
  - 原生侧根据系统设置选择 material 与 alpha；Flutter 侧避免过度半透明叠层并增强边界对比。

---

### D. Windows 11：Fluent 窗口材质（Mica 常驻 + Acrylic 临时）+ DWM 属性
**目标**：先把“窗口材质/圆角/暗色标题栏”等系统能力落地，再逐步扩展到原生 command bar/sidebar 与 channel。

#### D1. Runner 中设置系统 backdrop（Mica/Acrylic）
修改文件：
- `windows/runner/win32_window.cpp`（优先在 `UpdateTheme()` 或窗口创建后调用）

实现要点：
- 新增 `ApplyBackdrop(BackdropKind kind)` 封装：
  - 常驻：设置 `DWMWA_SYSTEMBACKDROP_TYPE` 为 `DWMSBT_MAINWINDOW`（Mica）
  - 临时：设置为 `DWMSBT_TRANSIENTWINDOW`（Acrylic）
- 保留并联动 `DWMWA_USE_IMMERSIVE_DARK_MODE`（暗色标题栏）
- 可选：设置 `DWMWA_WINDOW_CORNER_PREFERENCE`（圆角偏好），与 WinUI 3 常见圆角一致（克制 4–8dp 量级；窗口本身也有圆角）

参考（官方文档）：
- `DWM_SYSTEMBACKDROP_TYPE`：https://learn.microsoft.com/zh-cn/windows/win32/api/dwmapi/ne-dwmapi-dwm_systembackdrop_type
- `DWMWINDOWATTRIBUTE`（包含 `DWMWA_SYSTEMBACKDROP_TYPE`）：https://learn.microsoft.com/zh-cn/windows/win32/api/dwmapi/ne-dwmapi-dwmwindowattribute

#### D2. “临时 Acrylic” 的使用策略（第一阶段先做能力，不强依赖 channel）
- 先在 runner 内部提供切换 API（便于后续通过 channel 触发）。
- 第二阶段再把“弹窗/右键菜单/侧边栏”等 Flutter 侧事件通过 channel 通知 runner 切换 backdrop（避免第一阶段复杂度过高）。

#### D3. ui-ux-pro-max（Windows）落地点
- hover/focus：在桌面 common 的 Flutter 壳层先统一（不要把 Windows hover 逻辑散落到业务页）。
- 对比度：Mica 背景上确保文字与 focus ring 达到可读性；必要时加半透明底/分割线增强层次。
- 减少动效：尊重 `MediaQuery.disableAnimations`（切换页面/侧栏动画降级）。
